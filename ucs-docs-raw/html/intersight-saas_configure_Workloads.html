Workloads

Overview
Overview

Workloads provide a mechanism for end users to define a collection of configurations that can be applied consistently and repeatedly across a group of servers, devices, or clusters managed collectively as a fleet. Workloads are the top-level component of fleet groups.

For more information on fleet management, see:

  • Overview of Fleet Management

  • Supported Hardware Platforms

  • Prerequisites

  • Scale and Limits

  • Fleet Management Process

  • Tasks Involved in Fleet Management

Navigate to Configure > Workloads to open the Workloads Table View. This page shows three tabs: Definitions, Deployments, Instances.

  • Definitions—Workload definition is a golden configuration for a fleet of systems. It combines one or more blueprints with custom settings to ensure consistent deployment and management. For more information, see Definitions.

  • Deployments—A workload deployment is derived from a workload definition and applies the configuration to a set of target devices. For more information, see Deployments.

  • Instances—An instance represents the actual application of a configuration to a specific endpoint device as defined by a deployment. For more information, see Instances.

Creating Definition

Creating New Version of Definition
Creating New Version of Definition

After a workload definition version is deployed, further changes to its configuration parameters are not supported. To implement changes, a new version of the workload definition must be created. For the initial deployment, only the first version is considered.

To create a new version of a workload definition:

  1. Select a workload definition in the table view, click the ellipses (...), and choose Manage Definition Versions.

  2. On the selected Workflow Definitions page, click Create a New Version.

  3. In the dialog box, the “Source Version” and the “New Version” are displayed. The source version represents the base version from which the new version will be created.

  4. Click Create. The Create a New Version of Definition screen for the selected workload is displayed. Follow the steps to create a new definition.

    For details, see the Creating Definition section.

  5. While saving, you can set this new definition version as the preferred version by checking the "Set as Preferred Version" check box.

The following parameters cannot be changed in subsequent definition versions after the initial version is created.

  • On the General page:

    • Organization

    • Name

    • Platform Type

  • On the Blueprint Selection page, new blueprints cannot be added, and following parameters cannot be changed for existing blueprints:

    • OS Configurations

    • Constraints

    Note:

    • If you change the server firmware version, the upgrade will take effect on the next reboot by default, unless you select the "Immediate" option in the Server Firmware Options drop-down.

    • The maximum number of versions that can be created per definition is 10.

Configuring Workload Blueprints
Configuring Workload Blueprints

Cisco-published, validated workload blueprints are default standardized templates used to build workload definitions and contain all of the automation for server and operating system configuration. They can be used as part of Definitions.

The following Cisco-published validated blueprints are available:

  • Bare Metal Server—Bare Metal Server blueprint is Cisco Unified Edge bare metal optimized for AI and edge workloads. It provides only the hardware configuration without installing and configuring an operating system. It is ideal for IT teams looking to deploy their own operating system with maximum flexibility and reliability.

  • ESXi Server —VMware ESXi server blueprint is optimized for deploying VMware's virtualization platform and hypervisor.

  • Red Hat Server—Red Hat Enterprise Linux (RHEL) server blueprint is optimized for AI workloads and Kubernetes-based deployments with OpenShift integration.

  • Ubuntu Server—Ubuntu server blueprint is optimized for AI inferencing and containerized workloads, ensuring high efficiency and low latency in edge deployments.

  • Windows Server—Microsoft Windows server blueprint is optimized for deploying Microsoft Windows operating system and hypervisor platform.

Deployments
Deployments

A workload deployment is derived from a definition and applies the configuration to a set of target devices.

After creating a workload definition, one or more workload deployments can be created either immediately or at a later stage.

With a workload deployment, the following can be performed:

  • Specify unique configuration parameter overrides from the base definition.

  • Define localized maintenance windows for automatic upgrades.

  • Create qualification criteria to automatically match resources for deployment.

Creating Schedule Policy
Creating Schedule Policy

A Schedule policy defines the maintenance windows for workload deployments. It ensures that deployments or updates occur only during the specified windows. Schedule Policies apply exclusively to workload deployments.

Follow these steps to create a Schedule policy.

Before you begin: You must have Account Administrator, Workload Administrator, or Workload Operator privileges.

  1. Choose Configure > Policies, and click Create Policy.

  2. Select Schedule Policy and click Start to begin configuring the policy.

  3. On the General page, configure the following parameters:

    Property

    Description

    Organization

    Select the Organization.

    Cadence defines how often the scheduled deployment runs, with recurrence configurable in minutes, hours, days, weeks, or months. For each option, select a value within the allowed range. The system will automatically run the deployment at the specified interval.

    • Minutes – Select a value between 15–120 minutes

    • Hourly – Select a value between 1–24

    • Daily – Select a value between 1–30

    • Monthly – Select a value between 1–12 and specify a day or week of the month, and optionally a day of the week.

Updating Deployment
Updating Deployment

To apply configuration changes to an existing deployment, proceed with these steps:

  1. Create a New Definition Version.

    After a workload definition version is deployed, its configuration parameters cannot be changed. To make changes, you must create a new version of the workload definition.

    For more information, see Creating New Version of Definition.

  2. Test the New Version.

    Move a selected deployment to the new definition version and deploy it. This allows you to test the changes in isolation, without affecting other deployments using the same definition.

    For more information, see Creating Deployment.

  3. Mark as Preferred Version.

    Once you have tested and validated the changes in the test deployment, mark the new version as the preferred version.

    For more information, see Manage Definition Versions in Deployments Table View.

  4. Update Other Deployments.

    Deployments with the new version set as the preferred version will receive updates during the next change window.

Deployment Results
Deployment Results

During deployment, all automation associated with the blueprints in the workload are executed.

  • Groups of instances run according to the selected rollout strategy. The deployment process continues until all instances are complete, which may require multiple change windows. For more information, see Deployments Table View.

  • The system prepares Unified Edge templates and server templates. Then, it creates the required policies and profiles needed to run the configurations defined in the blueprints.

  • The profiles are deployed, and the operating system is installed on the servers.

  • Zero-touch provisioning is performed for new resources that qualify based on the Resource Qualification criteria. For more information, see Creating Deployment.

  • You can view the underlying policies, profiles, and templates created by the workload. For more information, see Unified Edge Profiles, Policies, Templates and Server Profiles, Policies, and Templates.

  • Actions such as editing, unassigning, or modifying these policies and profiles are not allowed, as that can impact the entire deployment.

  • You can safely update an individual profile or policy by first testing changes on a single resource and, once validated, applying those changes to the entire deployment by following these steps:

    1. Detach the instance from the deployment. For more information, see Instances Table View.

      The associated profiles will then become editable.

    2. Modify the necessary policies.

    3. Optionally, deploy the updated profile to the specific server for isolated testing.

    4. Once you have validated the changes, reattach the instance. For more information, see Instances Table View.

    5. Select the corresponding deployment from the Deployments Table View.

    6. Update the deployment by making the same changes you applied in Step 2, then edit and save the deployment configuration.

      This deployment will remain in the "Changes Scheduled" state until the next maintenance window.

    This approach allows you to minimize risk and ensure stability before propagating updates across the deployment.
