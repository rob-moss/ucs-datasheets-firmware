Deploying Nutanix Compute-Only Cluster on Cisco UCS Servers

Overview

Modern workloads require modern flexibility. Traditional hyperconverged solutions requires scaling compute and storage together. When you need to scale compute capacity without adding storage, Cisco Compute-Only (CO) nodes provide that critical capability, empowering you to design clusters that match your exact requirements.

The Cisco compute-only node with Nutanix Cloud Platform (NCP) and Pure Storage FlashArray solution delivers a disaggregated infrastructure architecture that enables independent scaling of compute and storage resources. This solution combines industry-standard Cisco compute-only node servers running Nutanix AHV and AOS connected to external Pure Storage FlashArray. This architecture leverages Pure software-defined storage environment optimized for enterprise workloads. The FlashStack solution with Nutanix integration provides organizations with the agility to adapt infrastructure resources dynamically while maintaining enterprise-level reliability and performance.

Key Benefits

  • Independent Scalability: Scale compute and storage resources separately based on workload demands

  • Enterprise-Grade Protection: Built-in data protection and disaster recovery capabilities

  • Hybrid Cloud Ready: Seamlessly extend operations across on-premises and cloud environments

  • Investment protection – enables using Nutanix Cloud Platform with existing external storage

Supported Network Topologies IMM
Supported Network Topologies IMM

With Pure Storage FlashArray Network Connectivity

  • Connect FabricInterconnects to top-of-rack (ToR) uplink switches using vPC port-channels, trunking all management, guest VM and storage VLANs.

  • Connect ethernet links to the FlashArray, one per controller to each ToR switch. Switch ports should be in access mode using two VLANs, one for the "A" side interfaces and one for the “B” side interfaces.

  • Links can be 25 GbE or 100 GbE connected to add-in ethernet cards.

  • Configure all ports on the ToR uplink switches to support jumbo frames.

With VPC (Virtual Port Channel) or MCT (Multi Channel Trunking)

This topology is recommended with either Cisco Nexus 9000 Series switches in VPC or any standard top of rack (ToR) Ethernet switch in MCT (multi-chassis trunking) mode.

  • The Ethernet uplink ports on the fabric interconnect can be used for Ethernet uplinks to the redundant ToR (top of rack) Nexus switches.

  • Port channels connect from the fabric interconnect to the ToR Nexus switches in VPC mode, or to a different ToR Ethernet switch in a multi-channel trunk.

  • ToR switch ports should be STP (spanning-tree) edge ports.

  • This topology:

    • Provides redundancy for the fabric interconnect, the ToR switches, and uplinks.

    • Provides bandwidth aggregation to the fabric interconnect through port channels.

    • Avoids ToR-to-ToR L2 switching.

  • The ToR fabric could be 3-tier, EVPN-VxLAN, an IP Fabric, or a CLOS leaf and spine topology.

With non-VPC (Virtual Port Channel) or non-MCT (Multi Channel Trunking)

This topology is recommended if your deployment does not use VPC or MCT. This topology is supported, but not recommended, if your deployment uses VPC or MCT. Instead, the previous topology is recommended for deployments that use VPC or MCT.

  • The Ethernet uplink ports on the fabric interconnect can be used for Ethernet uplinks to the redundant ToR (top of rack) Nexus switches.

  • In this topology:

    • vNICs are pinned to one uplink, and distributed across multiple uplinks.

    • ToR switches and uplinks are redundant, but vNICs will experience repinning time after a failover.

No Top of Rack Switch Redundancy

This topology conserves the number of overall links and ports, which can be used for additional server connectivity. This topology is supported, but not recommended, because of the lack of redundancy for the ToR Ethernet switches.

For more information on Cisco VIC 15000 Series Best Practices, please refer to Cisco VIC 15000 Series Best Practices in Ethernet Fabric White Paper.

Prepare for Cluster Deployment Using Intersight SaaS
Prepare for Cluster Deployment Using Intersight SaaS

Intersight connectivity is required for cluster deployment on servers managed with Intersight SaaS.

  • Before installing a Nutanix Compute-Only cluster on a set of Cisco servers, make sure that the device connector on the corresponding Fabric Interconnect instance is properly configured to connect to Cisco Intersight. Intersight has a minimum requirement of 1.5 Mbps of bandwidth and no greater than 500 ms latency to the device connector.

  • All device connectors must properly resolve svc.intersight.com or svc.eu-central-1.Intersight.com and allow outbound initiated HTTPS connections on port 443.

  • During cluster deployment, a bootstrap image is downloaded from Intersight to the servers to prepare for node imaging. To improve latency and overall download speeds, the URLs cdn.intersight.com or cdn.eu-central-1.intersight.com on port 443 should be reachable from the Fabric Interconnects for IMM servers.

Generate an Intersight User API Key
Generate an Intersight User API Key

An Intersight API key is required for cluster deployment in Nutanix Foundation Central. To generate an API key, do the following:

  1. Log in to Intersight with account administrator or server administrator privileges.

  2. Choose Settings > Keys > Generate API Key.

  3. Provide a description for the API key.

  4. Select API key for OpenAPI schema version 3.

  5. Set an Expiration Time if desired. Nutanix Prism Central needs a valid API authentication for continued cluster deployment.

  6. Click Generate.

  7. Copy the API key ID and Secret Key to a text file or download the Secret Key file and save it securely.

Note:

This is your only opportunity to view, copy and download the Secret/Private key. Intersight does not store the private key, and the private key cannot be recovered. However, you can create new API keys at any time.

  3. In Device ID, enter the applicable device ID.

    Note:

  4. In Claim Code, enter a target claim code. You can find this code in the Device Connector for the target type.

    Note:

  5. (Optional) To assign the server to a desired organization, select the appropriate resource group(s).

Users can be assigned to a role which grants access to a specific set of organizations. The user is only able to see entities and resources owned by the organizations to which they have access.   6. Click Claim.

  7. Refresh the Targets page to view the newly claimed target.

Configure the IMM Domain
Configure the IMM Domain

Before proceeding with cluster deployment, the IMM domain must be configured with a domain profile. This applies the desired networking configuration to the FIs and allows the servers to be discovered. For more information on configuring and deploying a domain profile, see Configuring UCS Domain Profiles.

Required Policies

The following policies must be configured in the domain profile to ensure that the network configuration required for cluster deployment is applied to the domain.

VLAN Policy and Multicast Policy

A VLAN policy must be defined and attached to each FI in the domain profile. The policy must have VLANs defined for the host and CVM traffic and can include additional VLANs for user VM traffic. To automatically add the VLAN to uplink ports on the FI, select the “Auto Allow On Uplinks“ option. A multicast policy must also be attached to each VLAN. Multiple VLANs can share the same multicast policy.

Port Policy

A Port policy must be defined and attached to each FI in the domain profile. The policy must have at least one port configured as an Ethernet uplink to the uplink switch. Ethernet network group, flow control, and link control policies are optional. If “Auto Allow On Uplinks“ is disabled for a given VLAN, then the Ethernet network group policy is required to ensure that traffic from that VLAN can traverse the uplink. Every port connected to a server chassis must be configured as a server port. If configuring a HCI X-Direct domain, server ports do not need to be configured.

System QoS Policy

A System QoS policy must be defined and attached to the domain profile. During Nutanix cluster deployment, the "Best Effort" QoS class is used, and vNICs are configured with an Ethernet QoS policy MTU of 9000. Therefore, the MTU for the "Best Effort" System QoS policy class should be set to 9216 to accommodate protocol overhead.

Network Connectivity Policy

A network connectivity policy must be defined and attached to the domain profile. The provided DNS server must be able to resolve svc.intersight.com and cdn.intersight.com to connect to Intersight and download the Phoenix OS image during cluster deployment.

NTP Policy

An NTP policy must be defined and attached to the domain profile. This ensures that the domain is synced with a time server and maintains an accurate system time.

Domain Profile Deployment and Server Discovery

Once the domain profile is created, assigned, and attached to the required policies, it must be deployed to apply the configuration. Once domain profile deployment is initiated, it can be monitored from the requests tab in Intersight. Once complete, the connected chassis will be discovered automatically, along with the installed servers.

Required Pools

Cluster deployment requires that at least one MAC address pool and IP address pool is configured. Optionally, a UUID pool can be configured to assign specific UUIDs to the nodes during profile association. All pools must belong to the same Intersight organization as the nodes.

MAC Address Pool

A MAC address pool must be configured prior to cluster deployment. This is required to assign MAC addresses to the vNICs during server profile association. The MAC address pool must contain enough addresses to accommodate two vNICs per adapter, per server.

IP Address Pools

At least one IP address pool must be configured prior to cluster deployment. This pool is used to derive the management IP addresses for each server during profile deployment. The pool must have enough addresses for each server in the cluster. E.g., if you intend to configure an inband and out-of-band IP address for each server, the pool must be able to accommodate allocating two IP addresses per server. A separate pool for inband and out-of-band IPs can also be specified, in which case each pool must be able to allocate one address per server.

UUID Pool

A UUID pool should be configured prior to cluster deployment if configuring custom UUIDs for each server is desired. The UUID pool must contain enough IDs to assign one UUID per server.

Foundation Central Support for Multiple vNICs
Foundation Central Support for Multiple vNICs
Foundation Central 1.10 introduces the capability to create multiple virtual Network Interface Cards (vNICs) on Cisco Virtual Interface Cards (VICs) for Integrated Management Module (IMM) clusters, with no support for ISM clusters. This feature supports both rack and modular servers connected to Fabric Interconnects. The maximum number of vNIC pairs that can be created is 10 pairs (20 vNICs), including the Infrastructure vNIC pair.

  • Supports multiple vNICs per Cisco VIC, leveraging Cisco SingleConnect technology for unified LAN, SAN, and management traffic.

  • Enables policy-based, stateless server infrastructure management.

  • Facilitates network traffic segregation and redundancy through multiple vNICs.

  • Integrates with Cisco Intersight for centralized management and advanced policy application.

  • This functionality enhances network flexibility and scalability in hyperconverged infrastructure deployments managed via Cisco Intersight and Foundation Central.

Modes of Operation

Simple Mode

  • Provides a user-friendly interface within Foundation Central.

  • Allows users to add vNIC pairs directly from the Foundation Central UI.

  • Results in a common LCP creation which is attached to all the server profiles in the cluster.

  • Simplifies the initial network configuration process. For details, refer to Nutanix Field Installation Guide.

Advanced Mode

  • Enables selection of an existing LAN Connectivity Policy (LCP) pre-created in Cisco Intersight.

  • Supports advanced configuration options such as Quality of Service (QoS).

  • The selection of the LCP is provided within Foundation Central.

  • Designed for users requiring granular control over vNIC settings.

Overview of the Profile and Policies of a Deployed Cluster
Overview of the Profile and Policies of a Deployed Cluster
Note:

We recommend against manually modifying these policies and profiles unless it is required and carefully planned. Modifying policies without a thorough understanding may cause cluster wide outages.

Server Profile Creation and Deployment

A server profile is created for each node. The profile is created with a unique name in the format ntnx_<serial>_<deployment uuid>, where <serial> is the serial number of the server and <deployment_uuid> is the deployment uuid generated by Nutanix Foundation Central (FC). The profile has all the applicable policies for that node attached to it. As part of the cluster deployment process, these profiles are activated to apply the policies to the servers.

Server Policies

Server policies particular to UCS servers with Nutanix Compute-Only clusters with Nutanix fall under four categories: per-cluster policies, per-configuration policies, node-specific policies, and default policies.

  • Per-cluster policies—Per-cluster policies are policies where a single policy instance applies to all nodes in the cluster since the configuration is shared regardless of the node type or model. The names of these policies are prefixed with the cluster name.

  • Per-configuration policies—Per-configuration policies are policies that differ based on the hardware configuration of the node. These policies have a prefix with the cluster name and a label describing the configuration.

  • Node-specific policies—Node-specific policies must have a separate policy instance for each node as the configuration is unique per server. These policies have a prefix with the serial number of the node.

  • Default policies—Default policies define configuration that is not required for Nutanix cluster operation but is created to ensure uniform configuration across nodes and to conform with Cisco best practices.

Per-cluster Policies

LAN Connectivity Policy (IMM only)—The LAN connectivity policy is configured as a per-cluster policy on IMM servers. The LAN Connectivity policy defines the vNICs on the VIC adapters of the server. Each adapter has two vNICs created, one for each fabric.

IMC Access Policy (IMM only) —A single IMC access policy instance is created per cluster. This policy configures the management IP address for the servers, which can be used for various management functions such as vKVM console access and SNMP trap messages.

Ethernet Adapter and Ethernet QoS Policy—These policies are attached to a given vNIC instance contained within the LAN Connectivity Policy. The Ethernet adapter policy controls host Ethernet interface settings, such as TCP offload and interrupt settings. The Ethernet QoS policy sets the MTU value of the vNIC.

  • Ethernet Network Group and Ethernet Network Control Policy (IMM only) - These policies are attached to a given vNIC instance contained within the LAN Connectivity Policy. The Ethernet Network Group policy determines the VLANs allowed on the associated vNIC. The Ethernet Network Control policy controls the behavior of the link during failures and enables discovery protocols.

Configuration-specific Policies

BIOS policy—Certain node hardware configurations may have different required BIOS settings, so this policy may be different depending on the node platform.

Storage Policy—A storage policy is created for each node to configure a VD called “NTNXOSBoot“ on the M.2 RAID controller. This VD is used to install the hypervisor during node imaging. This policy is only applicable if M.2 raid controller is installed in the system.

Node-specific Policies

Boot Order policy—Depending on the hardware of the node, the boot order precedence may be different across nodes in the same cluster. The boot target for the hypervisor is set first, followed by the CIMC-mapped FlexMMC target. B-series nodes will have a CIMC-mapped DVD boot target instead of FlexMMC. Secure boot is also enabled.

Storage Policy—A storage policy is created for each node to configure a VD called “NTNXOSBoot“ on the M.2 RAID controller. This VD is used to install the hypervisor during node imaging. This policy is only applicable if M.2 raid controller is installed in the system.

Firmware Policy—A firmware policy is created on each node. This ensures the node is running the latest qualified server firmware as part of server profile deployment during the HCI cluster installation process.

Default Policies

The following policies define configurations that are not required for Nutanix cluster operation but are still created to ensure a uniform configuration across nodes and to conform with Cisco best practices.

IPMI Over LAN Policy—IPMI over LAN is not required, so it is explicitly disabled in the policy. After cluster deployment, the user can choose to enable it if desired.

Serial Over LAN Policy—Serial over LAN is not required, so it is explicitly disabled in the policy. After cluster deployment, the user can choose to enable it if desired.

Virtual KVM Policy—Virtual KVM is not required for cluster deployment but should still be enabled in accordance with best practices. It must have the default settings of port 2068, max session count of 4, and tunneled KVM set to off. Video encryption and local server video are enabled with best practices.

Expansion Procedure

To expand a Nutanix HCI cluster:

  1. Prepare the nodes to be expanded

    :

    1. Use Nutanix Foundation Central (FC) to discover the nodes claimed in Intersight. For instructions, see Run Foundation Central for Deploying Hardware Provider Nodes.

    2. After the servers are claimed in Intersight, start the node preparation using Nutanix Foundation Central. For details, refer to the Nutanix Field Installation Guide.

      • Use the same AOS and hypervisor versions that are running on the existing cluster during the node preparation. An FC error will occur if the versions are different.

      • The server firmware version bundled with FC will be used for node preparation. This may cause the expansion node to have a different server firmware version than the nodes in the target cluster.

      • FC automatically creates the required Intersight policies and profiles to match the existing cluster configuration. This helps avoid manual errors and surprises during the expansion operation on a production cluster.

  2. Expand the cluster:

    1. Use Nutanix Prism Central (PC) to select the existing cluster and add the nodes prepared in the previous step.  For instructions, see Expanding a Cluster through Prism Central. Note: Expansion is faster than the initial deployment since the new nodes are prepared using the existing cluster configuration.

    2. Nutanix PC adds the node to the cluster and configures the cluster properties.

Add a new VIC to an existing IMM cluster

  1. In Nutanix Prism, put the node in Maintenance Mode.

    Note: 1-node clusters cannot be put in maintenance mode. Shut down the VMs before continuing to the next step.

  2. De-commission the server.

    Note: De-commissioning also unassigns the server profile.

  3. Install the new VIC.

    For instructions, refer to the server installation and maintenance guide. For example:

    • Cisco UCS X210c M7 Compute Node Installation and Service Guide

    • Cisco UCS C-Series Rack Servers Install and Upgrade Guides

  4. Re-commission the server and verify the new VIC inventory.

  5. Click on Server Profile and edit the LAN Connectivity policy.

  6. Add two vNICs for the new VIC adapter:

    • Clone the vNICS from existing vNIC: one on Switch ID A and one on B.

    • Use the default Auto vNICs Placement.

  7. Deploy and Activate the server profile.

    • If the newly added adapter firmware is lower, the server profile should show inconsistent state for Firmware Policy Out of Sync. This will be corrected after the server policy is deployed and activated.

  8. After the server boots to hypervisor, select Exit Node from Maintenance using Prism.

  9. For AHV, refer to the following Nutanix article to bring up the new NIC interfaces: KB 10029-How to configure newly added / replaced NIC on AHV host.

  10. Repeat steps 1 to 9 on all nodes.

  11. Add the new interfaces to the existing vSwitch or create a new vSwitch.

Upgrade a Cluster using Nutanix LCM

You can upgrade Nutanix compute clusters using the Nutanix Life Cycle Manager (LCM). LCM upgrade the following 2 software layers by determining any software dependencies, intelligently prioritizing the updates, and automating the entire upgrade process across all clustered hosts. This allows the upgrade to be performed efficient and reliable without impacting applications or data availability.

LCM upgrades the following 3 software layers:

  • Hypervisor (Nutanix AHV)

  • Hyperconverged software (Nutanix AOS)

Upgrade Limitations

  • Nutanix LCM cannot be used to upgrade server firmware on Compute-Only clusters.

Replace M.2 boot drives

If one M.2 boot drive fails on a server, complete the following steps to replace them. For both the M.2 drive failure, refer to the Nutanix documentation.

Replace a single drive

  1. Identify the slot of the failed M.2 drive.

  2. From Prism, put the node in maintenance mode.

  3. Gracefully power down the server.

  4. Remove the server power and network cables.

  5. Replace the drive. For details, see the "Replacing a Cisco Boot-Optimized M.2 RAID Controller" in the Installation and Service Guide for your UCS M6 or M7 server.

  6. Reinsert the power and network cables.

  7. Power on the server.

  8. Wait for the RAID virtual drive to be rebuilt.

  9. An alarm will be present on CIMC and Intersight as the drive is rebuilding.

  10. The alarm is automatically cleared after the controller rebuilds the data on the new drive.

  11. Exit the server from maintenance mode.
