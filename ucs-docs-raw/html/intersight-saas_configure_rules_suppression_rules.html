Granular Alarm Suppression Rules

Intersight provides a comprehensive framework for managing alarms generated by managed targets. While alarms are essential for alerting you to failures or threshold breaches, large-scale environments can often produce an overwhelming volume of notifications. To address this, Intersight offers advanced control over alarm visibility and email notification management through Granular Alarm Suppression Rules. These rules enable you to selectively suppress specific alarms, ensuring that only the most actionable issues demand your attention.

How Granular Suppression Rules Work

Granular Alarm Suppression Rules allow you to define logical criteria—such as alarm type, severity, or other properties—to automatically suppress selected alarms. When an alarm meets the conditions specified in a rule:

  • Email Notifications: The alarm will not trigger an email notification.

  • Intersight UI: The alarm is hidden from the main Alarms tab and individual device views. Instead, it is moved to the Suppressed tab, helping to declutter your interface and focus on relevant alerts.

Suppressed alarms are excluded from global alarm counts and do not affect device health indicators. For example, if a critical alarm is suppressed, the device will continue to display as "Healthy" (Green) unless other non-suppressed alarms are present.

Granular rules provide targeted control, preventing notification overload in large-scale environments. By hiding or muting only specific, non-critical alarms, administrators can streamline workflows and ensure operational focus remains on actionable events.

In addition to Granular Alarm Suppression Rules, Intersight offers two other mechanisms for managing alarm notifications: Email Notification Rules and Device-Level Suppression. Email Notification Rules allow you to configure which alarms trigger email alerts based on custom criteria.

Device-Level Suppression enables you to temporarily mute alarm notifications for individual devices, which is particularly useful during maintenance or planned activities.

For more information, see Email Notification Rules and Alarm Suppression.

Suppression Rules List View

When you navigate to Configure > Alarm Rules > Suppression Rules in the Intersight UI, the Suppression Rules list view appears.

The list view shows each rule in a tabular format. The table includes the following key details:

  • Name—The unique name assigned to the suppression rule.

  • Description— A brief description for the rule.

  • Status—The administrative state of the rule (Enabled or Disabled).

  • Conditions—Displays the specific filter criteria (properties, operators, and values) defined for the suppression rule.

  • Start Date—The scheduled date and time when the suppression becomes active.

  • End Date—The scheduled date and time when the rule expires. If no end date is specified, the rule remains active indefinitely until manually disabled or deleted.

  • Last Updated— The last time the rule was updated.

Alarm Suppression Actions

On the Suppression Rules page, you can click the ellipsis (…) icon or use inline actions next to a rule to:

  • Edit—Modify the existing rule conditions or values.

  • Clone—Create a new rule by copying all fields from an existing one.

  • Enable/Disable— Turn the suppression rule on or off.

  • Delete—Permanently remove the rule.

Creating a Granular Alarm Suppression Rule

Alarm suppression rules are created by defining logical conditions that determine which specific alarms should be hidden. Each rule requires selecting a filter property, a logical operator, and a specific value to establish the suppression criteria.

Note: The Intersight processes suppression rules at a rate of approximately 250 alarms per second.

Suppression rules are built using a flexible expression builder consisting of:

  • Filter Option (Property): The specific field to be evaluated. Supported properties include: Affected Name, Affected Type, Severity, Code, Parent Name, Parent Type, Profile Name, and UCS Domain.

  • Operator: The logical condition applied to the property.

    • Equals (=): Matches entries where the field exactly matches the specified value (case-sensitive).

    • In: Matches entries where the field value matches any of multiple specified values.

  • Value: The value must correspond to the specific criteria for the selected property, such as a specific HostName or Code.

For example, with:

  • Filter Option: Code

  • Operator: Equals

  • Value: AdapterBackupImage

The suppression rule will hide all alarms that are triggered for devices where the Code exactly matches "AdapterBackupImage".

To configure granular alarm suppression rule, perform the following steps:

  1. Log into Intersight as a user with an Account Administrator role.

  2. Navigate to Create Suppression Rule using either the Configure > Alarm Rules > Create Suppression Rules path or the Alarm drawer > Create Rule > Create Suppression Rule..

  3. Click Create Suppression Rule.

  4. In the Name field, type a unique string to identify the rule.

  5. In the Description field, optionally type a brief description.

  6. Select the Start Date and End Date for the suppression period.

    Note:

    If end date is not specified, the rule remains active indefinitely until manually disabled or deleted.

  7. Use the Add Filter field to select a filter option from the available built-in options. The filter option can be one of the following: Affected Name, Affected Type, Severity, Code, Parent Name, Parent Type, Profile Name, and UCS Domain.

  8. After selecting a filter option, specify the operator to define the logical condition for the filter. The available operators are:

    • Equals (=): Matches entries where the value in the selected field exactly matches the specified value.

    • ln: Matches entries where the field value is one of multiple specified values.

  9. Enter the value for the filter condition. The value must correspond to the selected filter option and operator.

  10. Repeat these steps to specify additional conditions.

    Note:

    You can specify a total of five filter conditions.

  11. If you specify multiple filters, use the Alarms will be suppressed when field to choose whether alarms should be suppressed when Any condition is met or only when All conditions are satisfied.

  12. Click Preview.

  13. Before saving, use the Preview pane to validate your configuration by sampling live matches. This pane automatically refreshes as conditions are modified to display the first 20 matches across all alarm states—Active and Acknowledged—prioritizing the newest alarms to demonstrate which records will be suppressed.

    Additionally, the Preview shows the total number of alarms matched to the rule, reflecting how many alarms correspond to the specific rule.

  14. Click Create.

Suppressed Alarms

When a granular suppression rule suppresses an alarm, that alarm is removed from the main Alarms tab and individual device views to help declutter your interface and allow you to focus on relevant alerts. Instead, the suppressed alarm is relocated to the Suppressed tab. To view these suppressed alarms, click on the icon and then select the Suppressed tab.

Configuring Granular Alarm Suppression Rules via API

This section explains how to configure alarm suppression rules using the Intersight APIs. Alarm suppression enables selective hiding of alerts across managed objects (MOs), ensuring only relevant alarms are visible and actionable.

Alarm suppression rules are defined using an array of conditions in the AlarmRules property. When multiple rules are specified, the RulesOperator property determines whether all conditions must be met (All for AND logic) or if any single condition is sufficient (Any for OR logic).

To create a granular alarm suppression rule with custom criteria, submit a POST request as shown below:

POST /api/v1/cond/AlarmSuppressions

In the example below, the suppression rule named "Suppression1" will be valid and active only between the specified StartDate and EndDate, suppressing alarms with the code "AssetTargetDisconnectedDC" for that window.

{
  "Name": "Suppression1",
  "StartDate": "2026-01-21T16:20:00-08:00",
  "EndDate": "2026-01-21T16:30:00-08:00",
  "RulesOperator": "All",
  "AlarmRules": [
    {
      "Property": "Code",
      "Operator": "eq",
      "Value": [
        "AssetTargetDisconnectedDC"
      ]
    }
  ]
}

To suppress alarms based on additional criteria such as severity, profile, or domain, add more objects to the AlarmRules array using the appropriate API property keywords mapped from the UI filter options and adjust the RulesOperator as needed.

  • Ensure correct mapping of UI filter options to API property keywords for accurate rule creation.

  • Multiple criteria can be combined using All (AND) or Any (OR) logic.

  • StartDate is always required; EndDate is optional for indefinite suppression.

  • Suppression is only enforced when the rule is enabled.
