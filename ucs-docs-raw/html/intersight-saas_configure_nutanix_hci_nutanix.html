Overview

Cisco Compute Hyperconverged with Nutanix is a hyperconverged infrastructure (HCI) solution integrating Cisco’s best-in-class Unified Computing System (UCS), datacenter networking, and SaaS Intersight infrastructure management platform with market-leading hyperconverged software from the Nutanix Cloud Platform. This solution supports Nutanix cluster configuration in Intersight for:

  • Cisco HCI X-Series Modular Server with Intelligent Fabric Module (IFM) or Fabric Interconnect Module (X series Direct) in Intersight Managed Mode (IMM).

  • Cisco HCI standalone rack servers in Intersight Standalone Mode (ISM) and Intersight Managed Mode (IMM).

  • A mix of modular servers and rack servers is supported in the same Nutanix cluster, but they must be connected to the same pair of Fabric interconnects. This mix is not supported when using HCI-X Direct.

The Cisco HCI X-Series delivers performance, flexibility, and optimization for deployments in data centers, in the cloud, and at remote sites. This enterprise-class server offers market-leading performance, versatility, and density without compromise for workloads. Up to eight hyperconverged nodes can reside in the 7-Rack-Unit (7RU) Cisco Compute Hyperconverged X9508 Chassis, offering one of the highest densities of compute, I/O, and storage per rack unit in the industry. Cisco Compute Hyperconverged X-Series Direct module simplifies your data center, adapting to the unpredictable needs of modern applications while also providing an edge scaled for remote branch office workloads. It minimizes the IT infrastructure deployed at edge locations to achieve desired business outcomes.

Supported Cisco HCI servers claimed in Intersight can be deployed as a Nutanix cluster with policies and profiles defined in Intersight. The supported HCI servers can be added to an existing UCS-X 9508 chassis. HCI servers can co-exist with non-HCI servers in the same chassis. The co-engineered integration between Nutanix Prism Central and Cisco Intersight enables seamless zero-touch remote deployment of Nutanix clusters with Nutanix AHV or VMware ESXi hypervisors.

All the required policies for node identity, network connectivity, BIOS, boot order, and more are automatically built based on Nutanix and Cisco recommended best practices. The resulting server profile is associated with each HCI server. After gathering the cluster configuration details, Nutanix Foundation Central (FC) orchestrates the end-to-end remote cluster deployment through Cisco Intersight without any manual intervention.

Note:

Nutanix Foundation Central (FC) is a service component of Nutanix Prism Central.

Cisco Compute Hyperconverged with Nutanix software delivers pre-configured Cisco HCI servers that are ready to be deployed as nodes to form Nutanix clusters in a variety of configurations. Each server contains three software layers:

  • UCS server firmware

  • Hypervisor (Nutanix AHV or VMware ESXi)

  • Hyperconverged software (Nutanix AOS)

Cisco Intersight provides integration support with Nutanix Prism Central to deploy Nutanix HCI clusters on Cisco UCS rack mount servers and X-series modular servers. This brings the HCI into remote edge and branch offices (ROBO) with cluster size starting from as low as one node and enterprise data centers for maximum scale. The solution provides maximum flexibility with choice of either a Cisco VIC or an Intel NIC for network connectivity leveraging existing network infrastructure without the need for additional networking gear.

This document covers prerequisites, supported network topologies and post-installation operations to:

  • Deploy Nutanix clusters in Intersight.

  • Expand existing clusters.

  • Upgrade clusters using Nutanix Life Cycle Manager (LCM).

Supported Network Topologies ISM
Supported Network Topologies ISM

Cisco Compute Hyperconverged with Nutanix on Cisco HCI rack servers in Intersight Standalone Mode supports 10G+ VIC or NIC network speed with dual Top-of-Rack (ToR) switch options for ultimate network flexibility and redundancy. A network topology is chosen during initial deployment and requires careful planning and execution for future changes or upgrades. Choose your network topology carefully and with future needs in mind.

10GE+ VIC-Based Topology

The 10GE+ VIC with dual Top-of-Rack (ToR) topology provides a fully redundant design that protects against switch, link, and port failures. The dual ToR switches can operate in standalone mode, stacked mode, or in spine-leaf architecture.

Cisco IMC Connectivity for 10GE+ VIC-Based Topology

The dedicated 1GE Cisco IMC management must be connected to the ToR switch and the IMC must be configured in dedicated mode. This option requires additional switch ports and cables. However, it avoids network contention and ensures always on, out-of-band access to each physical server.

You must assign an IPv4 management address to the Cisco IMC following the procedure in the Server Installation and Service Guide for the equivalent Cisco UCS C-series server.

Physical Network and Cabling for 10GE+ VIC-Based Topology

We recommend two managed switches with VLAN capability. We fully test and provide reference configurations for Catalyst and Nexus switching platforms. Choosing one of these switches provides the highest level of compatibility and ensures a smooth deployment and seamless ongoing operations.

This topology provides full redundancy that protects against switch failure, link failure, and port failure. It requires two switches that may be standalone or stacked, and at least two 10GE+ ports, one 1GE port for CIMC management, and one Cisco VIC per server.

ToR Port and VLAN Configuration

The ToR switch port can be configured as either a trunk port or an access port. We recommend using the trunk port configuration. See the following guidelines on port mode configuration and VLAN configuration.

  • Trunk Mode—The switch port must be configured in trunk mode with appropriate VLANs allowed. In this mode, management traffic from the cluster nodes is tagged with the VLAN ID. Additional VLANs are allowed in the switch port for additional network creation. If you choose this option, make sure to enter the management VLAN ID in the VLAN section of Nutanix Foundation Central cluster deployment wizard.

  • Access Mode—In this mode, the switch port must be configured in access mode with the management VLAN configured as native VLAN. If you choose this option, the management traffic from each node is untagged. This can be used when you are deploying the cluster in a flat network without any VLAN isolation. If you choose this option, you do not need to enter the VLAN ID in the VLAN section of Nutanix Foundation Central cluster deployment wizard.

Note:

If you are using a non Nexus switch, configure the ToR port in trunk mode even though you require the traffic untagged with VLAN. By default, traffic is not tagged by the hypervisor. However, it is tagged with VLAN ID 0 by the Cisco VIC. To make sure traffic remains untagged when it reaches the ToR port, configure the ToR port in trunk mode.

Requirements for Nutanix ISM clusters using 25GE Networking

Certain models and lengths of Cisco 25 GbE cables require the use of FEC set to mode CL74 for the link to become active. Cisco VIC cards have their external interfaces set to FEC mode CL91 by default, also known as FEC Auto. Some connected switches may also need their Ethernet interfaces manually set to CL74 to match for the link to become active. The FEC setting on the Cisco VIC external Ethernet interfaces can be manually changed via the CLI, but these settings are overridden by FC during the installation. In the FC cluster creation wizard, make sure to configure the appropriate FEC mode so that the interfaces are active and online after the server profile configuration is applied in Cisco Intersight. If the correct setting is not applied and the links are not active during the installation, the installation will fail.

10GE+ Dual Port VIC Topology

In this topology, you must have at least one dual port VIC installed in the mLOM slot of each node. Before starting the cluster deployment, make sure you have the following upstream network requirements.

  • Two managed switches (standalone or stacked).

  • 2 x 10GE+ ports and 1 x 1GE port for each HCI node.

  • Portfast trunk configured on all ports to ensure faster Spanning Tree Protocol (STP) convergence in case of any link or port failures.

  • From each server, connect the first port of the VIC to the first ToR.

  • From each server, connect the second port of the VIC to the second ToR.

    Note:

    Using the same port numbers in each ToR for a specific node helps management and troubleshooting. For example, if you connect server 1 VIC port 1 to ToR 1 port 5, make sure to connect server 1 VIC port 2 to ToR 2 port 5.

  • (Optional) Configure jumbo frames.

10GE+ Quad Port VIC Topology

In this topology, you must have at least one quad port VIC installed in the mLOM slot of each node. Before starting the cluster deployment, make the following upstream network requirements are met.

  • Two managed switches (standalone or stacked).

  • 4 x 10GE+ ports and 1 x 1GE port for each HCI node.

  • Portfast trunk configured on all ports to ensure faster Spanning Tree Protocol (STP) convergence in case of any link or port failures.

  • From each server, connect the first two ports (port 1 and 2) of the VIC to the first ToR.

  • From each server, connect the second two ports (3 and 4) of the VIC to the second ToR.

    Note:

    Using the same port numbers in each ToR for a specific node helps management and troubleshooting. For example, if you connect server 1 VIC port 1 & 2 to ToR 1 port 5 & 6, make sure to connect server 1 VIC port 3 & 4 to ToR 2 port 5 & 6.

  • (Optional) Configure jumbo frames.

Additional VICs

Adding up to two additional VIC cards for guest VM networking is supported in Intersight Standalone Mode. These can be installed prior to cluster deployment. However, keep them disconnected during cluster deployment. After the cluster is deployed, these VICs can be connected to ToR switches and configured as needed for guest VM networking.

Note:

Adding Intel NICs for additional networking in a node or cluster with a Cisco VIC is not supported.

Supported Network Topologies IMM
Supported Network Topologies IMM

With VPC (Virtual Port Channel) or MCT (Multi Channel Trunking)

This topology is recommended with either Cisco Nexus 9000 Series switches in VPC or any standard top of rack (ToR) Ethernet switch in MCT (multi-chassis trunking) mode.

  • The Ethernet uplink ports on the fabric interconnect can be used for Ethernet uplinks to the redundant ToR (top of rack) Nexus switches.

  • Port channels connect from the fabric interconnect to the ToR Nexus switches in VPC mode, or to a different ToR Ethernet switch in a multi-channel trunk.

  • ToR switch ports should be STP (spanning-tree) edge ports.

  • This topology:

    • Provides redundancy for the fabric interconnect, the ToR switches, and uplinks.

    • Provides bandwidth aggregation to the fabric interconnect through port channels.

    • Avoids ToR-to-ToR L2 switching.

  • The ToR fabric could be 3-tier, EVPN-VxLAN, an IP Fabric, or a CLOS leaf and spine topology.

With non-VPC (Virtual Port Channel) or non-MCT (Multi Channel Trunking)

This topology is recommended if your deployment does not use VPC or MCT. This topology is supported, but not recommended, if your deployment uses VPC or MCT. Instead, the previous topology is recommended for deployments that use VPC or MCT.

  • The Ethernet uplink ports on the fabric interconnect can be used for Ethernet uplinks to the redundant ToR (top of rack) Nexus switches.

  • In this topology:

    • vNICs are pinned to one uplink, and distributed across multiple uplinks.

    • ToR switches and uplinks are redundant, but vNICs will experience repinning time after a failover.

No Top of Rack Switch Redundancy

This topology conserves the number of overall links and ports, which can be used for additional server connectivity. This topology is supported, but not recommended, because of the lack of redundancy for the ToR Ethernet switches.

Configure the IMM Domain
Configure the IMM Domain

Before proceeding with cluster deployment, the IMM domain must be configured with a domain profile. This applies the desired networking configuration to the FIs and allows the servers to be discovered. For more information on configuring and deploying a domain profile, see Configuring UCS Domain Profiles.

Required Policies

The following policies must be configured in the domain profile to ensure that the network configuration required for cluster deployment is applied to the domain.

VLAN Policy and Multicast Policy

A VLAN policy must be defined and attached to each FI in the domain profile. The policy must have VLANs defined for the host and CVM traffic and can include additional VLANs for user VM traffic. To automatically add the VLAN to uplink ports on the FI, select the “Auto Allow On Uplinks“ option. A multicast policy must also be attached to each VLAN. Multiple VLANs can share the same multicast policy.

Port Policy

A Port policy must be defined and attached to each FI in the domain profile. The policy must have at least one port configured as an Ethernet uplink to the uplink switch. Ethernet network group, flow control, and link control policies are optional. If “Auto Allow On Uplinks“ is disabled for a given VLAN, then the Ethernet network group policy is required to ensure that traffic from that VLAN can traverse the uplink. Every port connected to a server chassis must be configured as a server port. If configuring a HCI X-Direct domain, server ports do not need to be configured.

System QoS Policy

A System QoS policy must be defined and attached to the domain profile. During Nutanix cluster deployment, the "Best Effort" QoS class is used, and vNICs are configured with an Ethernet QoS policy MTU of 9000. Therefore, the MTU for the "Best Effort" System QoS policy class should be set to 9216 to accommodate protocol overhead.

Network Connectivity Policy

A network connectivity policy must be defined and attached to the domain profile. The provided DNS server must be able to resolve svc.intersight.com and cdn.intersight.com to connect to Intersight and download the Phoenix OS image during cluster deployment.

NTP Policy

An NTP policy must be defined and attached to the domain profile. This ensures that the domain is synced with a time server and maintains an accurate system time.

Domain Profile Deployment and Server Discovery

Once the domain profile is created, assigned, and attached to the required policies, it must be deployed to apply the configuration. Once domain profile deployment is initiated, it can be monitored from the requests tab in Intersight. Once complete, the connected chassis will be discovered automatically, along with the installed servers.

Required Pools

Cluster deployment requires that at least one MAC address pool and IP address pool is configured. Optionally, a UUID pool can be configured to assign specific UUIDs to the nodes during profile association. All pools must belong to the same Intersight organization as the nodes.

MAC Address Pool

A MAC address pool must be configured prior to cluster deployment. This is required to assign MAC addresses to the vNICs during server profile association. The MAC address pool must contain enough addresses to accommodate two vNICs per adapter, per server.

IP Address Pools

At least one IP address pool must be configured prior to cluster deployment. This pool is used to derive the management IP addresses for each server during profile deployment. The pool must have enough addresses for each server in the cluster. E.g., if you intend to configure an inband and out-of-band IP address for each server, the pool must be able to accommodate allocating two IP addresses per server. A separate pool for inband and out-of-band IPs can also be specified, in which case each pool must be able to allocate one address per server.

UUID Pool

A UUID pool should be configured prior to cluster deployment if configuring custom UUIDs for each server is desired. The UUID pool must contain enough IDs to assign one UUID per server.

Foundation Central Support for Multiple vNICs
Foundation Central Support for Multiple vNICs

Foundation Central 1.10 introduces the capability to create multiple virtual Network Interface Cards (vNICs) on Cisco Virtual Interface Cards (VICs) for Integrated Management Module (IMM) clusters, with no support for ISM clusters. This feature supports both rack and modular servers connected to Fabric Interconnects. The maximum number of vNIC pairs that can be created is 10 pairs (20 vNICs), including the Infrastructure vNIC pair.

  • Supports multiple vNICs per Cisco VIC, leveraging Cisco SingleConnect technology for unified LAN, SAN, and management traffic.

  • Enables policy-based, stateless server infrastructure management.

  • Facilitates network traffic segregation and redundancy through multiple vNICs.

  • Integrates with Cisco Intersight for centralized management and advanced policy application.

  • This functionality enhances network flexibility and scalability in hyperconverged infrastructure deployments managed via Cisco Intersight and Foundation Central.

Modes of Operation

Simple Mode

  • Provides a user-friendly interface within Foundation Central.

  • Allows users to add vNIC pairs directly from the Foundation Central UI.

  • Results in a common LCP creation which is attached to all the server profiles in the cluster.

  • Simplifies the initial network configuration process. For details, refer to Nutanix Field Installation Guide.

Advanced Mode

  • Enables selection of an existing LAN Connectivity Policy (LCP) pre-created in Cisco Intersight.

  • Supports advanced configuration options such as Quality of Service (QoS).

  • The selection of the LCP is provided within Foundation Central.

  • Designed for users requiring granular control over vNIC settings.

Advanced Mode Requirements

The selected LCP must meet the following requirements:

  • The LCP must not be assigned to an existing server profile

  • The LCP must have a vNIC pair configured for Nutanix infrastructure traffic for each VIC adapter installed on a server. The vNIC pair must have the following configuration:

    • The vNIC names for the infrastructure pair must follow the convention ntnx-infra-1-A/ntnx-infra-1-B.

      • If more than one adapter is installed on the server, increment the digit in the third segment of the name for the subsequent vNIC pairs (e.g. ntnx-infra-2-A, ntnx-infra-3-A, and so on).

    • The fabric ID is must be set to A for ntnx-infra-1-A and B for ntnx-infra-1-B

    • Failover should be disabled.

    • VLANs specified in the Ethernet Network Group policy must contain the host and CVM VLAN.

    • MTU specified in the Ethernet QoS policy must match the MTU provided in Foundation Central.

  • It is strongly recommended to use the following Ethernet Adapter policy settings for all vNICs:

    • Select the Linux-v2 Cisco Provided Ethernet adapter configuration.

  • Enable etherchannel pinning, set transmit queue count to 2, and set completion queue count to 10.

Overview of the Profile and Policies of a Deployed Cluster
Overview of the Profile and Policies of a Deployed Cluster
Note:

We recommend against manually modifying these policies and profiles unless it is required and carefully planned. Modifying policies without a thorough understanding may cause cluster wide outages.

Server Profile Creation and Deployment

A server profile is created for each node. The profile is created with a unique name in the format ntnx_<serial>_<deployment uuid>, where <serial> is the serial number of the server and <deployment_uuid> is the deployment uuid generated by Nutanix Foundation Central (FC). The profile has all the applicable policies for that node attached to it. As part of the cluster deployment process, these profiles are activated to apply the policies to the servers.

Server Policies

Server policies particular to Cisco Compute Hyperconverged Infrastructure (HCI) with Nutanix fall under four categories: per-cluster policies, per-configuration policies, node-specific policies, and default policies.

  • Per-cluster policies—Per-cluster policies are policies where a single policy instance applies to all nodes in the cluster since the configuration is shared regardless of the node type or model. The names of these policies are prefixed with the cluster name.

  • Per-configuration policies—Per-configuration policies are policies that differ based on the hardware configuration of the node. These policies have a prefix with the cluster name and a label describing the configuration.

  • Node-specific policies—Node-specific policies must have a separate policy instance for each node as the configuration is unique per server. These policies have a prefix with the serial number of the node.

  • Default policies—Default policies define configuration that is not required for Nutanix cluster operation but is created to ensure uniform configuration across nodes and to conform with Cisco best practices.

Per-cluster Policies

LAN Connectivity Policy (IMM only)—The LAN connectivity policy is configured as a per-cluster policy on IMM servers. The LAN Connectivity policy defines the vNICs on the VIC adapters of the server. Each adapter has two vNICs created, one for each fabric.

Ethernet Adapter and Ethernet QoS Policy—These policies are attached to a given vNIC instance contained within the LAN Connectivity Policy. The Ethernet adapter policy controls host Ethernet interface settings, such as TCP offload and interrupt settings. The Ethernet QoS policy sets the MTU value of the vNIC.

  • Ethernet Network Policy (ISM only) - These policies are attached to a given vNIC instance contained within the LAN Connectivity Policy. This policy sets the trunking mode of the vNIC.

  • Ethernet Network Group and Ethernet Network Control Policy (IMM only) - These policies are attached to a given vNIC instance contained within the LAN Connectivity Policy. The Ethernet Network Group policy determines the VLANs allowed on the associated vNIC. The Ethernet Network Control policy controls the behavior of the link during failures and enables discovery protocols.

IMC Access Policy (IMM only)- A single IMC access policy instance is created per cluster. This policy configures the management IP address for the servers, which can be used for various management functions such as vKVM console access and SNMP trap messages.

Configuration-specific Policies

BIOS policy—Certain node hardware configurations may have different required BIOS settings, so this policy may be different depending on the node platform. For servers with NVMe drives and controllers, VMD is enabled for NVMe hot-plug support. PCH SATA option ROM is also disabled.

Node-specific Policies

Adapter Configuration Policy (ISM only)—An adapter configuration policy is created to control the adapter settings on all nodes. LLDP is enabled and port-channel is disabled on the VIC so that the adapter does not try to aggregate the two uplinks into a single virtual uplink.

Boot Order policy—Depending on the hardware of the node, the boot order precedence may be different across nodes in the same cluster. The boot target for the hypervisor is set first, followed by the CIMC-mapped FlexMMC target. Secure boot is also enabled.

Storage Policy—A storage policy is created for each node to configure a VD called “NTNXOSBoot“ on the M.2 RAID controller. This VD is used to install the hypervisor during node imaging.

LAN Connectivity Policy (ISM only)—The LAN connectivity policy is configured as a node-specific policy on ISM servers. The LAN Connectivity policy defines the vNICs on the VIC adapters of the server; if a VIC is not installed on the server, this policy is not configured. The LAN Connectivity policy has a single vNIC created for each physical port on the adapter.

Firmware Policy—A firmware policy is created on each node. This ensures the node is running the latest qualified server firmware as part of server profile deployment during the HCI cluster installation process.

Default Policies

The following policies define configurations that are not required for Nutanix cluster operation but are still created to ensure a uniform configuration across nodes and to conform with Cisco best practices.

SAN Connectivity Policy (ISM only)—FCoE connectivity is not required for Nutanix cluster configuration, but is created to ensure that each VIC adapter has the default vHBA configuration. As part of this policy, Fibre Channel Adapter, Fibre Channel Network, and Fibre Channel QoS polices are also created.

IPMI Over LAN Policy—IPMI over LAN is not required, so it is explicitly disabled in the policy. After cluster deployment, the user can choose to enable it if desired.

Serial Over LAN Policy—Serial over LAN is not required, so it is explicitly disabled in the policy. After cluster deployment, the user can choose to enable it if desired.

Virtual KVM Policy—Virtual KVM is not required for cluster deployment but should still be enabled in accordance with best practices. It must have the default settings of port 2068, max session count of 4, and tunneled KVM set to off. Video encryption and local server video are enabled with best practices.

Other Policies—The following policies are supported for standalone servers but are not used for Nutanix cluster deployment. Settings related to these policies that are already set on CIMC are retained after the profile is deployed, even if the policy is not present.

      In the Edit VIC Adapter Configuration Menu, disable the port-channel and appropriate FEC Mode on the interfaces.

        Example: For a Quad Port adapter, create 4 new vHBAs in SAN Connectivity Policy.

        Note: If the newly added adapter is of lower FW, the SP goes to inconsistent state for Firmware Policy Out of Sync. This will get corrected with Service Profile Deploy and activate, after making the above changes.

Replace M.2 boot drives

If one or more M.2 boot drives fails on a server, complete the following steps to replace them:

Replace a single drive

  1. Identify the slot of the failed M.2 drive.

  2. From Prism, put the node in maintenance mode.

  3. Gracefully power down the server.

  4. Remove the server power and network cables.

  5. Replace the drive. For details, see the "Replacing a Cisco Boot-Optimized M.2 RAID Controller" in the Installation and Service Guide for your UCS M6 or M7 server.

  6. Reinsert the power and network cables.

  7. Power on the server.

  8. Wait for the RAID virtual drive to be rebuilt.

  9. An alarm will be present on CIMC and Intersight as the drive is rebuilding.

  10. The alarm is automatically cleared after the controller rebuilds the data on the new drive.

  11. Exit the server from maintenance mode.

Replace both drives (for HCI only)

  1. Follow steps 2-7 in the previous procedure for single drive failure.

  2. After the server is powered on, deploy and activate the server profile assigned to the server. This will create the virtual drive on the new physical drives.

  3. In Prism, use the Repair Host Boot Device option to re-image the node.

  4. In Prism, run NCC checks on the cluster to ensure the cluster is operational after the repair process.

    Note: For both M.2 drive failure on CO node, refer to the Nutanix documentation.
